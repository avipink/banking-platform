## Domain Model

### Payment (Internal Entity)

**File**: `src/main/kotlin/com/digitalbank/payments/domain/Payment.kt`
**Type**: `data class` — internal only, never exposed at API boundary

| Field | Type | Source | Notes |
|---|---|---|---|
| `paymentId` | `String` | Internal | Generated by `PaymentRepository.nextId()` — format `PAY-NNN` |
| `fromAccountId` | `String` | `PaymentRequest` | Source (debited) account; validated via `AccountClient` |
| `toAccountId` | `String` | `PaymentRequest` | Destination (credited) account; validated via `AccountClient` |
| `amount` | `MonetaryAmount` | `PaymentRequest` | Value object from banking-contracts (`amount: String`, `currency: String`) |
| `type` | `PaymentType` | `PaymentRequest` | Enum from banking-contracts: `INTERNAL_TRANSFER`, `BILL_PAYMENT` |
| `status` | `PaymentStatus` | Set by service | Enum from banking-contracts: always `COMPLETED` on creation |
| `reference` | `String?` | `PaymentRequest` | Optional free-text memo; nullable |
| `createdAt` | `String` | `Instant.now().toString()` | ISO-8601 UTC timestamp set at creation time |

**Persistence**: None (in-memory mock). No JPA annotations. No database table.

## Repositories

### PaymentRepository

**File**: `src/main/kotlin/com/digitalbank/payments/repository/PaymentRepository.kt`
**Type**: `@Repository`, in-memory `MutableMap<String, Payment>`
**Persistence**: None — mock store. Not JPA-backed.

| Method | Signature | Description |
|---|---|---|
| `findAll()` | `(): List<Payment>` | Returns all payments |
| `findById(paymentId)` | `(String): Payment?` | Lookup by payment ID; returns null if not found |
| `findByAccountId(accountId)` | `(String): List<Payment>` | Returns all payments where `fromAccountId` or `toAccountId` matches |
| `save(payment)` | `(Payment): Payment` | Inserts or replaces payment in the map; returns saved payment |
| `getDailyTotal(accountId, today)` | `(String, String): BigDecimal` | Sums `amount.amount` for all outgoing payments where `fromAccountId == accountId` and `createdAt.startsWith(today)` |
| `nextId()` | `(): String` | Generates next sequential ID: `PAY-${map.size + 1}` formatted as 3-digit zero-padded string |

**Seeded Data**:
- `PAY-001`: ACC-001 → ACC-002, $500 USD, `INTERNAL_TRANSFER`, `COMPLETED`, `2026-02-28T10:15:00Z`
- `PAY-002`: ACC-003 → ACC-005, $1200 USD, `BILL_PAYMENT`, `COMPLETED`, `2026-02-25T14:30:00Z`
- `PAY-003`: ACC-002 → ACC-001, $250 USD, `INTERNAL_TRANSFER`, `PENDING`, `2026-03-01T08:00:00Z`

## HTTP Clients

### AccountClient

**File**: `src/main/kotlin/com/digitalbank/payments/client/AccountClient.kt`
**Target Service**: `accounts-core-svc`
**Base URL Config**: `${accounts-service.base-url}` (default: `http://localhost:8081`)

| Method | HTTP Call | Returns | Error Handling |
|---|---|---|---|
| `findAccount(accountId)` | `GET /api/v1/accounts/{id}` | `AccountResponse?` | Returns null on `WebClientResponseException.NotFound` (404) or any other exception |
| `isAccountActive(accountId)` | (delegates to `findAccount`) | `Boolean` | Returns false if `findAccount` returns null; checks `account.status == AccountStatus.ACTIVE` |

**HTTP Library**: Spring WebFlux `WebClient` — non-blocking reactive client called with `.block()` at the service boundary (synchronous invocation within a Spring MVC context).

## Exception Hierarchy

```
RuntimeException
  └── PaymentDomainException(val error: PaymentError)
        └── error is one of:
              PaymentError.InvalidAccount(accountId: String)
              PaymentError.InsufficientFunds(accountId: String)
              PaymentError.DailyLimitExceeded(limit: MonetaryAmount, attempted: MonetaryAmount)
```

`PaymentError` is a sealed class defined in `banking-contracts`. All three variants are handled exhaustively in `GlobalExceptionHandler` via a `when` expression.

---

## Error-to-HTTP Mapping

| PaymentError Variant | HTTP Status | ApiError.code |
|---|---|---|
| `InvalidAccount` | 404 | `PAYMENT_ACCOUNT_NOT_FOUND` |
| `InsufficientFunds` | 422 | `INSUFFICIENT_FUNDS` |
| `DailyLimitExceeded` | 422 | `DAILY_LIMIT_EXCEEDED` |

Note: `InsufficientFunds` is handled in `GlobalExceptionHandler` but never thrown by the current `PaymentService` — it is a forward compatibility stub from the `PaymentError` sealed class in banking-contracts.
